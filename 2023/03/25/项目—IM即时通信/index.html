消息传递流程：

  1. 假设 Alice上线，连接到机器[1]上：

    机器1将Alice和它的连接存入内存，调用user status的online方法记录Alice上线。

  2. Alice 发送一条消息给 Bob：

    1. 机器1收到消息后，解析destId，在内存中查找是否有Bob。
    2. 如果没有，代表Bob未连接到这台机器，则转发给transfer。

  3. transfer调用user status的getConnectorId(Bob) 方法找到Bob所连接的connector，返回机器[2]

总结：

引入user status模块管理用户连接，transfer模块在不同的机器之间转发，使服务可以水平扩展。为了满足实时转发，transfer需要和每台connector机器都保持长链接。

离线消息

如果用户当前不在线，就必须把消息持久化下来，等待用户下次上线再推送，这里使用mysql存储离线消息。

为了方便地水平扩展，可以使用消息队列进行解耦：

  • transfer接收到消息后如果发现用户不在线，就发送给消息队列入库；
  • 用户登录时，服务器从库里拉取离线消息进行推送。

离线消息表：

msg_type用于区分消息类型（chat,ack），content加密后的消息内容以byte数组的形式存储。

用户上线时按照条件to_user_id=用户id拉取记录即可。

  • 防止离线消息重复推送：

    1. 首先取出所有has_read=false的字段；

    2. 检查每条消息的has_read值是否为false，如果是，则改为true。这是原子操作。

    3. 修改成功则推送，失败则不推送。

可靠性

对于IM系统来说，可靠是指 不丢消息，消息不重复，不乱序。

  • 不丢消息： 加入ack机制，确认对方收到了消息。

    具体实现可以模仿tcp协议做一个应用层的ack机制：

    • tcp以字节为单位，而这里用message为单位。
    • 发送方每次发送一个消息，就要等待对方的ack回应，在ack确认消息中应带有收到的 id 以便发送方识别。
    • 发送方需要维护一个等待ack的队列。每次发送一个消息后，就将消息和一个计时器入队。
    • 在另外一个线程一直轮询队列，如果有超时未收到ack的，就去除消息重发。

    超时未收到ack的话，有两种处理：

    • 和tcp一样，不断发送知道收到ack为止；

    • 设定一个最大重试次数，超过这个次数还没收到ack，就使用失败机制处理，节约资源。

      例如如果是connector长时间未收到client的ack，那么可以主动断开和客户端的连接，剩下未发送的消息就作为离线消息入库，客户端断连后尝试重连服务器即可。

  • 不重复、不乱序：

    有的时候因为网络原因可能导致ack收到较慢，发送方就会重复发送，那么接收方必须有一个去重机制。

    去重的方式是给每个消息增加一个唯一id。这个唯一id只需要在一个会话中唯一即可，例如某两个人的会话，或者某一个群。如果网络断连了，重新连接后，就是新的会话了，id会重新从0开始。

    • 接收方需要在当前会话中维护收到的最后一个消息的id，叫做lastId。

    • 每次收到一个新消息， 就将id与lastId作比较看是否连续，如果不连续，就放入一个暂存队列 queue中稍后处理。

    还有一点，当客户端上线时，应该先推送完离线消息，再更新session。否则，先更新了session的话，在推送离线消息时，可能有新消息进来，导致乱序。

安全性

客户的隐私需要保护，因此所有的消息都必须加密处理。

可以将通信密钥保存在用户关系表relation中，当客户端登录时，就会从数据库中获取该用户所有的relation，存在内存中，以便后续加密解密。 当客户端给某个好友发送消息时，取出内存中该关系的密钥，加密后发送。同样，当收到一条消息时，取出相应的密钥解密。

项目描述

  • 该项目是一个即时聊天通信项目，用于和好友一对一聊天或加群群聊，以及传输文件。

技术栈： Springboot + Netty + Redis + RabbitMQ + MinIO

实现功能：

  1. 实现token方式的登录注册，避免跨域和session共享问题。

    登录时返回随机UUID作为token，redis存储token 【key: login:token:token值；value：用户简略信息】， 使用拦截器对所有页面进行token持续时间刷新，

  2. 实现点对点聊天，群聊，消息未读标记，上线消息推送，好友管理，好友在线查看等功能。（netty + redis + mysql）

    好友管理：

    1. 搜索用户
    2. 发送好友请求： 添加好友请求记录到数据库
    3. 接收好友请求： 查询数据库，有没有发给自己的好友请求
    4. 处理好友请求： 拒绝=直接删除数据库信息，接收=相互增加好友记录到双方数据库表
    5. 查询好友列表： 数据库查询即可

    聊天：

    有最新消息的聊天排在前面， 使用redis中SortedSet来存储会话列表（单聊群聊都一样）

    key：prefix_session_list:{sessionID}；聊天ID

    value：{chatCount}； 聊天信息总数

    score：{msgID}； 最新聊天信息ID

    用户对每一个聊天的进度，未读信息数 = chatCount - readCount；

    key：prefix_read:{uid}:{sessionID}； 用户所有的聊天

    value：{readCount}； 已读信息总数

    score：{msgID}； 已读最新聊天ID

    （新入群的成员不能看到入群前的信息，可以维护一个入群时的msgID，群中大于msgID的msg才能被看到）

    • 单聊： 收到消息，先保存到mysql数据库并标为“未签收”，如果接收者在线则直接发送，不在线则推送（未实现）。 接收者收到消息会发送ack，于是将此消息标为“已签收”。

    • 群聊：

      字段名 注释
      gid 群id
      uid 用户id
      last_ack_msgid 最后接收的消息id
      字段名 注释
      gid 群id
      msgid 消息id
      sender_uid 发送者id
      time 发送时间
      content 内容

      消息只存一份，使用读扩散。 用户刚加入群时，给当前群的最大msgid作为 最后接收的消息id，用户看群消息默认拉取最新30条消息，每次向上拉取都拿30条消息。

      • 新消息数量 = 群最大msgid - last_ack_msgid， 点击有x条新消息，直接拉取msgid > last_ack_msgid 的该群消息。

    好友在线查看：

    redis中存储用户在线信息： 【key：online_user:{uid}；value：1/0】

    登录上线时value置为1，退出或心跳读写都超时时置为0；

  3. 基于netty实现心跳机制，无读写时保持连线。利用RabbitMQ有序处理离线消息。

    netty心跳实现： IdleStateHandler设置读写超时时间，自定义的HeartBeatHandler()执行超时处理，如关闭channel，设置用户下线状态等

    上线推送：根据redis中已读的聊天信息ID和已读数，判断是否有新消息。

  4. 基于MinIO实现文件存储功能，支持断点续传。

    使用minIO进行分布式存储；

    断点续传： 在上传前先进行分块， 每个分块上传时标记上传中，上传完成后标记已上传。

MinIO

MinIO 适合存储大容量非结构化的数据，例如图片，视频，日志文件，备份数据和容器/虚拟机镜像等。

MinIO集群采用去中心化共享架构，每个节点是对等关系，通过Nginx可对MinIO进行负载均衡访问。

  • MinIO的去中心化和分布式，可以用于搭建一个高可用的对象存储服务： MinIO将分布在不同服务器上的多块硬盘组成一个对象存储服务。由于硬盘分布在不同的节点上，分布式MinIO避免了单点故障。
  • MinIO使用纠删码技术来保护数据。 纠删码是一种恢复丢失和损坏数据的数学算法， 它将数据分块冗余地分散在各个节点的磁盘上，所有可用磁盘组成一个集合，当上传一个文件时会通过纠删码算法计算 ，将文件进行分成n块，还会生成n个校验码，分散存储在n个硬盘上。 如果丢失了一半以下数量的硬盘，仍然可以恢复数据，不过在恢复重构时需要消耗CPU资源且有延时。 （类似 abc 分成了 ab，bc，ac，损失一半以下仍能恢复）

Session实现

session的工作原理：

  1. 打开浏览器，在浏览器上发送首次请求
  2. 服务器会创建一个HttpSession对象，该对象代表一次会话
  3. 同时生成HttpSession对象对应的Cookie对象，并且Cookie对象的name是jsessionid，Cookie的value是32位长度的字符串（jsessionid=xxxx）
  4. 服务器将Cookie的value和HttpSession对象绑定到session列表中
  5. 服务器将Cookie完整发送给浏览器客户端
  6. 浏览器客户端将Cookie保存到缓存中
  7. 当再次发送请求的时候，会自动提交缓存中当的Cookie
  8. 服务器接收到Cookie，验证该Cookie的name是否是jsessionid，然后获取该Cookie的value
  9. 通过Cookie的value去session列表中检索对应的HttpSession对象

基于Session的登录：

  • 发送验证码
    1. 校验手机号是否合法，不合法重新输入，合法就生成验证码；
    2. 将验证码保存到session；
    3. 发送验证码；
  • 登录、注册
    1. 用户提交手机号和验证码，后台从session中拿到当前验证码，进行校验，不一致则返回；
    2. 如果一致，则后台根据手机号查询用户，如果用户不存在，则为用户创建账号，保存到数据库。
    3. 保存用户信息到session中，方便后续获得登录信息。
  • 验证登录状态（再次访问，无需登录）
    1. 用户发起请求并携带cookie， 后台从session中获取用户信息；
    2. 判断用户是否存在，如果没有session信息，则进行拦截，
    3. 如果有session信息，则将用户信息保存到threadLocal中，并且放行。

session共享问题:

一般服务器会以集群的形式存在，即每次访问不一定是访问同一台服务器，当seesion信息只保存在上次登录的服务器中，所以访问到其他服务器时就会被拦截。

早期的解决方案是session拷贝，就是说虽然每个tomcat服务器上都有不同的session，但是每当任意一台服务器的session修改时，都会同步给其他的Tomcat服务器的session，这样的话，就可以实现session的共享了。

  • 缺点： 每台服务器都有完整的一份session数据，服务器压力过大。session拷贝数据时，可能会出现延迟

现在的解决方案：redis数据本身就是共享的，使用redis来完成信息保存。
